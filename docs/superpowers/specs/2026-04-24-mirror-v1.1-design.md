## Summary

Two features that wire existing Squad infrastructure to actually enforce policy at runtime:

1. **Token budget enforcement at claim time** — reject a claim if the squad wallet cannot cover the task's estimated cost.
2. **River policy key** — a Redis-backed governance document that lets River set per-squad fuel grades and token caps; sovereign reads it at dispatch before claim.

No new schema migrations. No new wallet models. All wiring into existing paths.

---

## Feature 1 — Token Budget Enforcement at Claim Time

### Problem

`SquadTaskService.claim()` (`tasks.py:504`) performs an atomic DB claim but never checks the squad wallet balance. The wallet debit happens at `complete()` (`tasks.py:690`) only. An agent can claim a task the squad wallet cannot afford — the wallet goes negative and no rejection ever fires.

`task.inputs["estimated_cost_cents"]` is already computed at create time by `estimate_task_budget()` and stored in `inputs_json`. The wallet table (`squad_wallets`) already tracks `balance_cents`. All the data is there.

### Fix

Inside `claim()`, after the task row is fetched (line ~520) and before the atomic UPDATE:

1. Read `estimated_cost_cents` from `task.inputs` (fallback: `FUEL_COST_CENTS.get(fuel_grade, 0)` × markup).
2. Read `squad_wallets.balance_cents` for `(squad_id, tenant_id)`.
3. If `balance_cents < estimated_cost_cents` → raise `InsufficientFundsError(task_id, balance_cents, estimated_cost_cents)`.

The check happens inside the same `with self.db.connect() as conn:` block before the UPDATE — no new connection, no TOCTOU gap.

```python
# After task row is fetched, before atomic UPDATE:
estimated_cents = task.inputs.get("estimated_cost_cents", 0)
if estimated_cents > 0:
    wallet_row = conn.execute(
        "SELECT balance_cents FROM squad_wallets WHERE squad_id = ? AND tenant_id = ?",
        (task.squad_id, tenant_id or DEFAULT_TENANT_ID),
    ).fetchone()
    balance = wallet_row["balance_cents"] if wallet_row else 0
    if balance < estimated_cents:
        raise InsufficientFundsError(
            task_id=task_id,
            balance_cents=balance,
            estimated_cost_cents=estimated_cents,
        )
```

`InsufficientFundsError` is a new exception in `tasks.py` (subclass of `ValueError` for backward compatibility). The HTTP handler in `app.py` maps it to `402 Payment Required`.

### Scope

- `sos/services/squad/tasks.py` — add `InsufficientFundsError`, add balance check in `claim()`
- `sos/services/squad/app.py` — add `except InsufficientFundsError → 402` handler on claim route
- `tests/test_task_budget_enforcement.py` — new test file, 3 tests:
  - claim succeeds when balance ≥ estimated cost
  - claim raises `InsufficientFundsError` when balance < estimated cost
  - claim succeeds when `estimated_cost_cents` is missing (no cost task)

## Feature 2 — River Policy Key

### Purpose

River needs a way to govern which fuel grades and token budgets apply per squad (or globally). Rather than hardcoding caps in the sovereign dispatcher, River writes a JSON document to Redis that sovereign reads before dispatch.

### Key

```
sos:policy:governance
```

Value: JSON document structured as:

```json
{
  "version": 1,
  "updated_at": "2026-04-24T00:00:00Z",
  "updated_by": "river",
  "global": {
    "max_fuel_grade": "aviation",
    "max_token_budget": 50000
  },
  "squads": {
    "<squad_id>": {
      "max_fuel_grade": "regular",
      "max_token_budget": 10000
    }
  }
}
```

Fuel grade ordering for cap enforcement: `diesel < regular < aviation < supernova`.

### Sovereign integration (dispatch path)

In `sovereign/loop.py` (or the dispatcher), before calling `claim()`:

1. Read `sos:policy:governance` from Redis (`GET`, parse JSON, cache for 60s in-process).
2. Look up squad-level policy (fall back to `global` if squad not present).
3. Cap `fuel_grade` to `min(task_fuel_grade, policy.max_fuel_grade)`.
4. Cap `token_budget` to `min(task.token_budget, policy.max_token_budget)`.
5. Update `task.inputs["fuel_grade"]` and `task.token_budget` before `claim()`.

If the key is missing or Redis is unavailable: proceed without caps (fail open, log a warning).

### Write-gate

Only a caller presenting River's bus token may write `sos:policy:governance`.

Implementation: a new restricted endpoint in the Squad Service (or SOS Engine):

```
PUT /policy/governance
Authorization: Bearer <river-token>
Content-Type: application/json
```

The handler:
- Validates the token is River's (check against `SOS_BUS_TOKENS` or a dedicated `RIVER_POLICY_TOKEN` env var).
- Validates the JSON schema (version, updated_at, global fields present).
- Writes to Redis with `SET sos:policy:governance <json>`.
- Readable by any authenticated caller (dispatcher, brain, monitoring).

Read endpoint:
```
GET /policy/governance
Authorization: Bearer <any-valid-token>
```

### Scope

- `sovereign/loop.py` — add policy read + cap injection at dispatch (with 60s in-process cache)
- `SOS/sos/services/squad/app.py` — add `PUT /policy/governance` + `GET /policy/governance` routes
- `SOS/sos/services/squad/policy.py` (new) — `GovernancePolicy` Pydantic model, `load_policy()`, `apply_caps()` helpers
- `tests/test_policy_governance.py` — 4 tests: write (River token), write (wrong token → 403), read, cap application logic

## Implementation Order

1. Feature 1 (budget enforcement) — Squad Service only, self-contained, no Redis dependency.
2. Feature 2 (River policy key) — depends on nothing, but touches sovereign dispatch and Squad Service.

Each ships as a separate PR, each gated by Athena.

---

## Out of Scope

- Retroactive wallet top-up or credit system (separate billing feature).
- Per-agent fuel grade caps (River policy is per-squad for now).
- Policy history or audit log (can add in v1.2 if needed).
